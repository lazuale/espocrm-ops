# V3

`v3` is the retained small stable backup/restore product exposed at the `espops` root.

## Mission

`v3` exists to keep the retained backup/restore product small and stable.

## Principles

- minimum code
- maximum backup/restore reliability
- no magic
- no guessing
- no legacy compatibility
- native tools first
- manifest is contract
- verify before trust
- snapshot before destructive restore
- success only after check

## Native Tooling Baseline

`v3` does not implement Docker or MariaDB.

`v3` calls native Docker Compose and MariaDB tools, then verifies artifacts and fails closed.

If a native tool fails, `v3` fails.
No fallback.
No guessing.
No masking.

### Allowed Docker Commands

- compose validation:
  `docker compose --env-file <env> -f <compose> config`
- service state:
  `docker compose --env-file <env> -f <compose> ps --format json`
- stop app services:
  `docker compose --env-file <env> -f <compose> stop <app-services...>`
- start app services:
  `docker compose --env-file <env> -f <compose> start <app-services...>`
- bring up db service:
  `docker compose --env-file <env> -f <compose> up -d <db-service>`
- execute in db service:
  `docker compose --env-file <env> -f <compose> exec -T <db-service> <command...>`
- minimal db post-check:
  `docker compose --env-file <env> -f <compose> exec -T <db-service> mariadb -u <user> <db> -e "SELECT 1;"`

### Allowed MariaDB Commands

- target native tooling:
  MariaDB `11.4 LTS`
- allowed backup:
  `mariadb-dump --single-transaction --quick --routines --triggers --events -u <user> <db>`
- allowed restore:
  `mariadb -u <user> <db>`
- password policy:
  use `MYSQL_PWD` or an already-existing native password file / secret mechanism
- password policy:
  never print the password
- password policy:
  never print a command line containing the password

### Allowed Archive And Checksum Tooling

- `.sql.gz`
- `.tar.gz`
- `.sha256`
- JSON manifest
- Go stdlib only:
  `compress/gzip`, `archive/tar`, `crypto/sha256`, `encoding/json`, `os`, `io`, `path/filepath`

### Forbidden Custom Implementations

- no Docker SDK abstraction
- no custom compose parser
- no custom service discovery engine
- no custom health orchestration framework
- no SQL dumper
- no SQL parser
- no MariaDB compatibility layer
- no custom archive format
- no custom checksum format
- no manifest framework beyond a small struct plus validation
- no resultbridge or output framework for `v3`
- no fallback to alternative tools
- no hidden repair
- no implicit service guessing

If required config is missing, fail.
If a Docker or MariaDB command fails, fail.
If dump or restore fails, fail.
If verify fails, fail.
Warnings do not replace failure.

### Minimal Go Wrapper Rules

- keep the native command wrapper small
- assemble command args explicitly
- use `exec.CommandContext`
- pass `stdin` / `stdout` only when needed
- include `stderr` in the returned error
- do not hide exit failures
- do not retry without an explicit reason
- do not add fallback paths
- do not guess service names
- keep `v3` glue code as command execution plus result verification

Preferred minimal shape:

```go
type DockerRuntime struct{}

func (DockerRuntime) ComposeConfig(ctx context.Context, target Target) error
func (DockerRuntime) Services(ctx context.Context, target Target) ([]Service, error)
func (DockerRuntime) Stop(ctx context.Context, target Target, services ...string) error
func (DockerRuntime) Start(ctx context.Context, target Target, services ...string) error
func (DockerRuntime) Up(ctx context.Context, target Target, service string) error
func (DockerRuntime) DumpDB(ctx context.Context, target Target, w io.Writer) error
func (DockerRuntime) RestoreDB(ctx context.Context, target Target, r io.Reader) error
func (DockerRuntime) DBPing(ctx context.Context, target Target) error
```

This is a direction constraint, not a requirement to add an interface layer where none is needed.

### What Not To Touch

- do not add a second runtime framework for `v3`
- do not add compatibility shims for legacy commands
- do not add fallback behavior to make native tool failures look successful
- do not expand this baseline into a custom backup engine
- do not introduce extra docs unless `V3.md` stops being enough

## Commands

Current:

- `espops doctor --scope <dev|prod>`
- `espops backup --scope <dev|prod>`
- `espops backup verify --manifest <path>`
- `espops restore --scope <dev|prod> --manifest <path>`
- `espops migrate --from-scope <dev|prod> --to-scope <dev|prod> --manifest <path>`

Not in first scope:

- partial backup
- latest lookup
- direct artifact restore/verify

## Backup-Set Contract

- manifest lives in `manifests/`
- backup artifacts live in sibling `db/` and `files/`
- sidecars use `.sha256`
- actual checksum, manifest checksum, and sidecar checksum must match
- `.sql.gz` and `.tar.gz` must be readable
- unsafe tar entries are rejected

## Anti-Goals

- no `resultbridge`
- no `contract/result`
- no shared transport framework
- no acceptance forest
- no oracle harness
- no architecture symmetry
- no compatibility shims

## Cleanup Policy

- delete dead code in separate PRs
- only delete what is demonstrably not used
- do not mix cleanup with feature PRs
- when in doubt, mark the candidate and defer
